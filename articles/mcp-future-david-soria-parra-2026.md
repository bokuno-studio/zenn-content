## MCPアプリケーション：インターフェースとしてのMCP

![MCPアプリケーションのデモ - ClaudeでRaspberry Pi図を描く](/images/screenshot-mcp-david-01.jpg)

冒頭のデモでは、**MCP サーバーがアプリそのものをシッピングする**という未来の姿が示されました。

従来の UI（プラグイン・SDK・クライアントレンダリング）ではなく、MCP サーバー経由でアプリを配信。そのサーバーを Claude、ChatGPT、VS Code、Cursor など **どのクライアントに挿しても動く**。

さらに MCP サーバーは UI だけでなく **ツールも一緒に配信**できます。ユーザーは UI として、モデルはツールとして、同一のサーバーと対話できる——これは他の方式では実現できない、MCP ならではの特性です。

## 2026年：本番エージェントのための接続性

David は年代を次のように整理します：

| 年 | エージェントの状況 |
|----|-------------------|
| 2024 | デモ・PoC が中心 |
| 2025 | コーディングエージェント全盛 |
| 2026 | 汎用エージェントが本番稼働へ |

コーディングエージェントは「ローカル・ファイアウォール内・コンパイラで検証可能・開発者が目の前にいる」という理想的な条件で動きます。しかし金融アナリストやマーケターが使う汎用エージェントに必要なのは別のものです：

**5つの SaaS アプリへの接続性**

## エージェントの3種類の接続方式

### 1. Skills（スキル）

エージェント固有の専門知識をシンプルなファイルで定義。最も再利用しやすい。

### 2. CLI

![CLIのUnixフィロソフィー](/images/screenshot-mcp-david-03.jpg)

ローカルエージェントに最適。Bash ツール合成・Progressive Discovery・事前学習済み CLI（gh、git など）との相性が良い。

- **Bash Tool Composition**: パイプ・リダイレクト・スクリプトで合成
- **Progressive Discovery**: 必要になったときに CLI の能力を探索
- **Pretraining**: gh、git など事前学習済みのツールに特に強い

### 3. MCP

リッチなセマンティクスが必要な場合・認可やガバナンスが必要な場合・サンドボックスなし環境・プラットフォーム非依存が求められる場合に選ぶ。

> 「もし誰かが一つの方法で接続の全問題を解決できると言ったら、それは間違いだ。正しい答えは常に『場合による』。」

## クライアント側で今すぐできること

### Progressive Discovery（段階的ツール発見）

多くの実装では **すべてのツールをコンテキストに一括ロード**しています。しかしこれはコンテキストウィンドウの肥大化を招きます。

代わりに：

```
ツール検索ツールを提供
→ モデルが「このツールが必要かも」と判断したときだけ呼び出す
→ オンデマンドでロード
```

Claude Code への適用で大幅なツールコンテキスト削減を実現。API や競合製品でも同様のパターンが使えます。

### Programmatic Tool Calling（プログラム的ツール呼び出し）

![プログラム的ツール呼び出しの例 - LinearからNotionへ](/images/screenshot-mcp-david-04.jpg)

モデルが「ツール呼び出し → 結果取得 → ツール呼び出し」と逐次処理するのは非効率です。

代わりに **REPL（実行環境）を与えてスクリプトを書かせる**：

```javascript
// 1回のREPL呼び出しで完結
const issue = await mcp.call_tool("linear_get_issue", { id: "ENG-5121" });
const page = await mcp.call_tool("notion_createPage", {
  parent: "Postmortems",
  title: issue.title,
  Body: issue.description
});
emit(page, { format: "json" });
```

MCP の **Structured Output** 機能で戻り値の型情報を活用すれば、モデルが型安全に複数ツールを合成できます。

## サーバー設計：エージェントファーストで考える

REST API を MCP に 1:1 変換するだけのサーバーは避けるべきです。エージェントのために設計する3原則：

1. **プログラム的ツール合成をサーバー側でも** — 実行環境を提供してモデルに合成させる
2. **MCP リッチセマンティクスを活用** — Tasks、Elicitations、MCP Applications など CLI では不可能なものを使う
3. **エージェントがどう使うかを軸に設計** — 人間向け REST の写しではなく、エージェントのユースケースを主体に

## 2026年の MCPロードマップ

![MCP 2026ロードマップ](/images/screenshot-mcp-david-05.jpg)

### コアの改善

| 改善項目 | 内容 |
|---------|------|
| **Stateless Transport** | Google が提案。大規模クラウドへのデプロイが容易になる（Cloud Run・Kubernetes 対応）。6月リリース予定 |
| **非同期タスクの改善** | エージェント間通信の基盤を整備 |
| **TypeScript SDK v2.0 / Python SDK v2.0** | 過去1年の学びをベースに大幅刷新。Python SDK は David 自ら「FastMCP の方がはるかに良い」と認める |

### どこでも統合できる

| 機能 | 内容 |
|------|------|
| **Cross-App Access** | 企業の ID プロバイダー（Google・Okta など）と連携。一度ログインすれば全 MCP サーバーにアクセス可能 |
| **Server Discovery** | ウェルノウン URL（`.well-known/mcp-server-card/server.json`）で MCP サーバーを自動発見。ブラウザ・エージェント・クローラーが対応 |

### 境界を押し広げる

| 機能 | 内容 |
|------|------|
| **Skills over MCP** | MCP サーバーにスキルファイルを同梱して配信。プラグイン機構やレジストリに依存せずスキルを更新できる |
| **MCP Applications** | Web ベースのインターフェースでのみ動作する拡張。今後も実験的機能を追加予定 |

**[0:00]** ようこそ。始めましょう。これが Anthropic のアプリケーションです。エージェントが出荷するのは「インターフェース」です——プラグインでも、SDK でも、モデルがクライアント側でレンダリングするものでも、ハードコードされた製品機能でもありません。MCP サーバー経由で提供されるものです。そのサーバーを Claude に、ChatGPT に、VS Code に、Cursor に入れれば、ただ動く。それは素晴らしいことだと思います。これを実現するためには、エコシステムの他の選択肢が持っていないものが必要です——セマンティクス、クライアントとサーバーの相互理解、UI のレンダリング方法の理解、UI が来ることの認識。そのためにプロトコルが必要です。そして最良の点は、MCP サーバーはアプリだけでなくツールも一緒に配信できるということです。人間がアプリケーションとして操作し、モデルはツールとして操作できる——これは非常にユニークで、まだほとんど探求されていない領域です。では少し巻き戻して、18ヶ月前に戻りましょう。AI のライフサイクルでは永遠に等しい時間です。そのころはすべてが存在せず、ほんの小さなスペック文書と数個の SDK があるだけで、ほぼ Anthropic 製のローカル専用・ツールのみの状態でした。この12〜18ヶ月でみなさんは驚くほど多くのサーバーを構築し、エコシステムを作り上げました。私たちもローカル専用からリモート機能・集中認可・新しいプリミティブ（Elicitation・Tasks）・そして MCP アプリケーションのような実験的機能の追加で忙しくしていました。そして月次ダウンロード数 1億1000万という素晴らしいマイルストーンに到達しました。React という最も成功したオープンソースプロジェクトの一つが同じ規模に達するのに要した時間の約半分です。

**[5:00]** 接続性について話しましょう。接続性は一つではありません。一つの方法で全接続問題を解決できると言う人がいれば、それはほぼ間違いです。正しい答えは常に「場合による」。2026年にエージェントを構築するとき、考慮すべき3つの主要要素があります：スキル・MCP・CLI（またはコンピューターユース）。スキルは主要な知識・特定の能力をシンプルなファイルに収めたもので最も再利用しやすい。CLI はローカルコーディングエージェントで非常に人気があり、Bash で閉じた単純な開始点として優秀。モデルが自動発見できて、gh・git など事前学習済み CLI との相性が良い。しかしリッチなセマンティクスが必要・長時間タスクの UI を表示したい・認可やガバナンスポリシーが必要・サンドボックスなし環境では MCP が適している。2026年は、これらを一つだけでなく**すべて組み合わせる**エージェントを作り始める年になります。今日特に伝えたいのは「**Progressive Discovery**」を構築し始めるべきだということです。現状の実験段階では単純にすべてのツールをコンテキストウィンドウに突っ込んでいますが、代わりに「ツール検索ツール」を提供してオンデマンドでロードするパターンを使うべきです。Claude Code への適用で大幅な削減を実現しました。

**[10:00]** 次は「**Programmatic Tool Calling**（プログラム的ツール呼び出し）」、別名「コードモード」です。モデルに逐次ツール呼び出しさせる代わりに、実行環境（V8 isolate、Deno、Lua インタープリターなど）を与えてコードを書かせ、そのコードが複数ツールを合成する方式です。Claude Code が Bash コマンドを書くのと同じ発想です。MCP の Structured Output 機能で戻り値の型情報を提供すれば、モデルが型安全に複数ツールを合成できます。「Linear のイシューを Notion ページにして」という一つのタスクを、2回の逐次ツール呼び出しではなく1回の REPL 呼び出しで完結できます。次に、サーバーを**エージェントファーストで設計**してください。REST API を MCP に1:1変換するだけのサーバーをいつも見るたびに少し残念に思います。MCP サーバー側でも実行環境を提供してモデルに合成させる。Claude コンピューターユース MCP サーバーなどが良い例です。そして Tasks・Elicitations・MCP アプリケーションなど CLI では実現できない MCP 固有の機能を活かしてください。

**[15:00]** MCP 自体に必要な改善も多くあります。まずコアの改善。現在の Streaming HTTP はハイパースケーラーにとってスケールが難しい。Google が提案する Stateless Transport Protocol で、Cloud Run や Kubernetes へのデプロイが簡単な通常のステートレスサーバーとして扱えるようになります——6月リリース予定。Python SDK v2.0・TypeScript SDK v2.0 も刷新します。FastMCP を使っている人がいますね？私が書いた Python SDK よりはるかに良い（笑）。より良い Python 開発者たちに手伝ってもらっています。次に**どこでも統合できる**機能。**Cross-App Access** は ID プロバイダー（Google・Okta 等）と連携して一度のログインで全 MCP サーバーにアクセスできる仕組み。**Server Discovery** はウェルノウン URL（`.well-known/mcp-server-card/server.json`）で MCP サーバーを自動発見できる仕組みで、6月の仕様更新と同時にリリース予定です。そして境界を押し広げる実験的機能として **Skills over MCP** をリリース予定。大量のツールを持つ MCP サーバーに、使い方の知識を同梱して配信できます。レジストリに依存せず継続的にスキルを更新できます。2026年はエージェントが完全な接続性を持つ年になります。MCP は引き続き主要な役割を果たします。フィードバックをください——Discord・Issue で何が間違っているか、何が正しいかを教えてください。最高のエージェントはコンピューターユース・CLI・MCP・スキル——あらゆる接続方法を使います。ありがとうございました。

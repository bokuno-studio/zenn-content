個人開発でClaude CodeとOpenAI Codexを両方使い始めて2ヶ月が経った。「なんとなく使ってる」ままにしたくなかったので、ちゃんとトークン消費量を計測してみた。3〜5月途中（5/7まで）の合計で**40億トークン超**、ピークの4月だけで**37.7億トークン**という数字が出てきたので記録として残しておく。

## 計測方法

**Claude Code**
Claude Codeは`~/.claude/projects/`配下にJSONL形式で会話ログを残す。各レコードにモデル名・タイムスタンプ・input/output/cache_creation/cache_readトークン数が含まれているので、これをパースして月次・モデル別に集計した。

**Codex**
Codexは`~/.codex/sessions/`配下にセッション単位でJSONLログを残す。ログ内に複数回出る`token_count`イベントのうち、最後の`total_token_usage`をそのセッションの使用量として集計した。

## 月次推移

| 月 | Claude トークン | Claude コスト | Codex トークン | Codex コスト | 合計コスト |
|---|---|---|---|---|---|
| 2026年3月 | 2.63億 | $115 | 0.02億 | $0.4 | ~$115 |
| 2026年4月 | **35.7億** | **$1,950** | **1.96億** | **~$147** | **~$2,100** |
| 2026年5月 ※ | 1.57億 | $112 | 0.60億 | ~$43 | ~$155 |

※ Claude / Codex ともに 2026-05-07 まで。

Claudeコストは API料金換算値。Codexコストはモデル別公開レートからの推定（gpt-5.3-codex: $1.75/$14 per M、gpt-5.4: $2.50/$15 per M、gpt-5.5: $5.00/$30 per M、cached inputは各10%）。

3月はClaude Codeを使い始めた時期で、レートリミットに何度も到達し、その都度プランを上げていた。4月に入りさらに使い込み始め、4月中旬ごろにClaudeのリミットへの分散先としてCodexを本格導入した。4月は両ツール合計で**37.7億トークン**と、3月の約14倍に跳ね上がった。

### 4月に何があったか

日次で分解すると、**4月13日の1日だけで2.2億トークン**を消費していた。

| 日付 | Claude Code トークン |
|---|---|
| 4/9 | 1,100万 |
| 4/10 | 1,480万 |
| 4/13 | **2億1,980万** ← |
| 4/14 | 409万 |
| 4/16 | 629万 |
| 4/20 | 2,337万 |
| 4/23 | 1,190万 |

この週（4/13週）だけで月全体の68%を占めている。内訳を見るとoutputが2.4億トークンの大部分を占めており、Agent/Subagentを大量に並列実行していた日と一致する。

### モデル別の内訳

**Claude Code（4月）**

| モデル | トークン | 推定コスト |
|---|---|---|
| claude-sonnet-4-6 | 22.4億 | $1,086 |
| claude-opus-4-7 | 7.4億 | $550 |
| claude-opus-4-6 | 2.0億 | $238 |
| claude-haiku-4-5 | 3.9億 | $76 |

ClaudeはSonnetをメインに使いつつ、4月から登場したOpus-4-7を即採用した。

**Codex（全期間）**

| モデル | 期間 | トークン | cached比率 | 推定コスト |
|---|---|---|---|---|
| gpt-5.3-codex | 3月 | 0.02億 | 80% | ~$0.3 |
| gpt-5.4 | 4月 | 0.69億 | 91% | ~$43 |
| gpt-5.5 | 4月下旬〜5月 | 1.89億 | 96% | ~$148 |

モデルの進化が速く、3月のgpt-5.3-codexから始まり4月頭にgpt-5.4、4月末にはgpt-5.5へ切り替わった。inputの94%がcached inputで、モデルが新しいほどキャッシュ比率が高い。gpt-5.5はトークン単価が高いぶん、キャッシュ効率で吸収する設計になっている。

## キャッシュが全体の95%以上（Claude Code）

Claude Codeのトークン内訳で驚いたのがキャッシュ読込の比率。

| 種別 | 3-4月合計トークン |
|---|---|
| input（生） | 342K |
| output | 17.5M |
| キャッシュ作成 | 144M |
| **キャッシュ読込** | **37.7億** |
| 合計 | **38.3億** |

全体の**96%がキャッシュ読込**。Claude Codeは会話のたびに大きなコンテキスト（CLAUDE.md、コードベース、会話履歴）をキャッシュ経由で再注入する設計になっているため、実際の生inputはごく少量で済んでいる。キャッシュ読込の料金はinputの1/10程度なので、実コストはさらに圧縮されている。

## 役割分担

2つのツールは競合ではなく、用途で使い分けている。Claude CodeはBiz系タスク（ドキュメント・調査・設計・資料整理）、CodexはDev（実装）担当だ。

トークン量に差があるのは用途の性質上の話で、Bizタスクはコンテキストが膨らみやすく会話も長くなる。加えてCodexは4月中旬からの本格稼働なので、期間も揃っていない。単純な量の比較に意味はあまりない。

## 所感

- **4月の爆発**はMulti-agent並列実行を本格的に試した時期と一致する。Subagentを10本同時に走らせるだけでトークンはあっという間に跳ね上がる
- **キャッシュ依存度が高い**ということは、大きなコンテキストを前提に設計されているということ。CLAUDE.mdや標準ドキュメントを充実させるほど効果が出る
- **Codexはモデル進化が速い**。3ヶ月で3世代変わった。gpt-5.5のcached input比率の高さはコスト的に有利
- **トークン構成の見方に注意**。Claude Codeはoutputとcache_readが大きく、Codexはcached inputが支配的。どちらも単純な生input/outputだけを見ると実態を見誤る

計測してみて、「なんとなく使ってる」の解像度が上がった。特にキャッシュ比率とAgent並列実行の影響は、意識していないと見えてこない部分だった。

個人開発でClaude CodeとOpenAI Codexを両方使い始めて2ヶ月が経った。「なんとなく使ってる」ままにしたくなかったので、ちゃんとトークン消費量を計測してみた。結果として**合計40億トークン超**という数字が出てきたので記録として残しておく。

## 計測方法

**Claude Code**
[ccusage](https://github.com/ryoppippi/ccusage) というCLIツールを使った。Claude Codeが`~/.claude/projects/`配下に残すJSONLファイルをパースして、モデル別・月次のトークン数を集計してくれる。

```bash
npx ccusage@latest
```

**Codex**
`~/.codex/sessions/2026/03-04/` に残るセッションログ（JSONファイル）の`token_count`フィールドを日付・モデル別に集計。

## Claude Codeの月次推移

| 月 | 合計トークン | API換算コスト | Max plan対比 |
|---|---|---|---|
| 2026年3月 | 2.63億 | $115 | 0.6x |
| 2026年4月 | **35.7億** | **$1,950** | **9.8x** |

3月はこわごわ使い始めた時期。4月に入ってから本格的に使い込み始め、**トークン消費が13.5倍に跳ね上がった**。

### 4月に何があったか

日次で分解すると、**4月13日の1日だけで2.2億トークン**を消費していた。

| 日付 | Claude Code トークン |
|---|---|
| 4/9 | 1,100万 |
| 4/10 | 1,480万 |
| 4/13 | **2億1,980万** ← |
| 4/14 | 409万 |
| 4/16 | 629万 |
| 4/20 | 2,337万 |
| 4/23 | 1,190万 |

この週（4/13週）だけで月全体の68%を占めている。内訳を見るとoutputが2.4億トークンの大部分を占めており、Agent/Subagentを大量に並列実行していた日と一致する。

### モデル別の内訳（4月）

| モデル | トークン | API換算コスト |
|---|---|---|
| claude-sonnet-4-6 | 22.4億 | $1,086 |
| claude-opus-4-7 | 7.4億 | $550 |
| claude-opus-4-6 | 2.0億 | $238 |
| claude-haiku-4-5 | 3.9億 | $76 |

4月からclaude-opus-4-7が登場し、即採用した。コスト的にはsonnetとopus-4-7で全体の84%を占める。

## キャッシュが全体の95%以上

Claude Codeのトークン内訳で驚いたのがキャッシュ読込の比率。

| 種別 | 3-4月合計トークン |
|---|---|
| input（生） | 342K |
| output | 17.5M |
| キャッシュ作成 | 144M |
| **キャッシュ読込** | **37.7億** |
| 合計 | **38.3億** |

全体の**96%がキャッシュ読込**。Claude Codeは会話のたびに大きなコンテキスト（CLAUDE.md、コードベース、会話履歴）をキャッシュ経由で再注入する設計になっているため、実際の生inputはごく少量で済んでいる。

逆にいうと、キャッシュが効いているからこそAPI換算でこれだけの「消費」が成立している。キャッシュ読込の料金はinputの1/10程度なので、実コストはさらに圧縮されている。

## Codexの使用量

| 期間 | モデル | inputトークン | outputトークン | セッション数 |
|---|---|---|---|---|
| 2026-03-06 | gpt-5.3-codex | 193K | 12K | 1 |
| 4月上旬〜中旬 | gpt-5.4 | 1.5億 | 155万 | 約90 |
| 4月下旬〜 | gpt-5.5 | 1.3億 | 40万 | 14 |
| **合計** | | **約1.97億** | **約124万** | **108** |

Codexのinputの特徴として、**93%がcached input**だった。gpt-5.5になるとこの比率がさらに上がる。

モデルの進化が速く、3月のgpt-5.3-codexから始まり、4月頭にgpt-5.4、4月末にはgpt-5.5へ切り替わった。

## Claude Code vs Codex の規模感

| ツール | 期間 | 合計トークン |
|---|---|---|
| Claude Code | 3〜4月 | **38.3億** |
| Codex | 3〜4月 | **1.98億** |
| **合計** | **3〜4月** | **約40.3億** |

**Claude Codeが約20倍のスケール**で消費している。これはツールの設計の違いによるところが大きい。

Claude Codeは対話型で1セッションあたりのコンテキストが巨大（CLAUDE.md + コードベース + 会話履歴）。一方Codexはタスク単位で完結するため1セッションのトークン消費は比較的コンパクト。

ただしCodexのセッション数108に対してClaude Codeは同期間で384セッション。セッション単位で割ると差はもう少し縮まる。

## Max planのコスパ

Claude CodeはMax plan（$200/月固定）で使っている。4月のAPI換算コストが$1,950だったので、

> **$1,950 ÷ $200 = 9.8倍のコスパ**

もちろんAPI換算コストは「もしAPIで同じことをやったら」という仮の数字で、実際にはMax planの制約（レート制限）があるため単純比較はできない。とはいえ、個人開発でここまで使い倒せるなら$200は安いと感じる。

3月や5月は0.6x相当なので、普段は余裕を持って使えているが、Agentを大量並列で走らせると一気に跳ね上がる。

## 所感

- **4月の爆発**はMulti-agent並列実行を本格的に試した時期と一致する。Subagentを10本同時に走らせるだけでトークンはあっという間に跳ね上がる
- **キャッシュ依存度が高い**ということは、大きなコンテキストを前提に設計されているということ。CLAUDE.mdや標準ドキュメントを充実させるほど効果が出る
- **Codexはモデル進化が速い**。3ヶ月で3世代変わった。gpt-5.5のcached input比率の高さはコスト的に有利
- **どちらもinput<<output**。AIが書き出すトークンの方が読み込みより圧倒的に多い。これはコーディング作業では必然かもしれない

計測してみて、「なんとなく使ってる」の解像度が上がった。特にキャッシュ比率とAgent並列実行の影響は、意識していないと見えてこない部分だった。

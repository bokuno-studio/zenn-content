個人開発でClaude CodeとOpenAI Codexを両方使い始めて2ヶ月が経った。「なんとなく使ってる」ままにしたくなかったので、ちゃんとトークン消費量を計測してみた。結果として**合計40億トークン超**という数字が出てきたので記録として残しておく。

## 計測方法

**Claude Code**
Claude Codeは`~/.claude/projects/`配下にJSONL形式で会話ログを残す。各レコードにモデル名・タイムスタンプ・input/output/cache_creation/cache_readトークン数が含まれているので、これをパースして月次・モデル別に集計した。

**Codex**
Codexは`~/.codex/sessions/`配下にセッション単位でJSONファイルを残す。ファイルの末尾に`token_count`オブジェクトがあり、`input`・`output`・`cached_input`・`reasoning_output`の内訳が記録されている。ファイル名のタイムスタンプ（Asia/Tokyo）とモデル名を使って日次・モデル別に集計した。

| 月 | Claude トークン | Claude コスト | Codex トークン | Codex コスト | 合計コスト |
|---|---|---|---|---|---|
| 2026年3月 | 2.63億 | $115 | 0.02億 | $0.4 | ~$115 |
| 2026年4月 | **35.7億** | **$1,950** | **1.96億** | **$454** | **~$2,400** |
| 2026年5月 ※ | 1.57億 | $112 | — | — | — |

※ Claude は5/7まで。Codexの5月分は集計対象外。

Claudeコストは API料金換算値。Codexコストはモデル別公開レートからの推定（gpt-5.3-codex: $1.75/$14 per M、gpt-5.4: $2.50/$15 per M、gpt-5.5: $5.00/$30 per M、cached inputは各50%）。

3月はClaude Codeを使い始めた時期で、レートリミットに何度も到達し、その都度プランを上げていた。4月に入りさらに使い込み始め、4月中旬ごろにClaudeのリミットへの分散先としてCodexを本格導入した。4月は両ツール合計で**37.7億トークン**と、3月の約14倍に跳ね上がった。

### 4月に何があったか

日次で分解すると、**4月13日の1日だけで2.2億トークン**を消費していた。

| 日付 | Claude Code トークン |
|---|---|
| 4/9 | 1,100万 |
| 4/10 | 1,480万 |
| 4/13 | **2億1,980万** ← |
| 4/14 | 409万 |
| 4/16 | 629万 |
| 4/20 | 2,337万 |
| 4/23 | 1,190万 |

この週（4/13週）だけで月全体の68%を占めている。内訳を見るとoutputが2.4億トークンの大部分を占めており、Agent/Subagentを大量に並列実行していた日と一致する。

## キャッシュが全体の95%以上

Claude Codeのトークン内訳で驚いたのがキャッシュ読込の比率。

| 種別 | 3-4月合計トークン |
|---|---|
| input（生） | 342K |
| output | 17.5M |
| キャッシュ作成 | 144M |
| **キャッシュ読込** | **37.7億** |
| 合計 | **38.3億** |

全体の**96%がキャッシュ読込**。Claude Codeは会話のたびに大きなコンテキスト（CLAUDE.md、コードベース、会話履歴）をキャッシュ経由で再注入する設計になっているため、実際の生inputはごく少量で済んでいる。

逆にいうと、キャッシュが効いているからこそAPI換算でこれだけの「消費」が成立している。キャッシュ読込の料金はinputの1/10程度なので、実コストはさらに圧縮されている。

| モデル | inputトークン | outputトークン | cached比率 | 推定コスト |
|---|---|---|---|---|
| gpt-5.3-codex（3月） | 193K | 12K | 80% | $0.4 |
| gpt-5.4（4月） | 6,783万 | 82万 | 91% | $105 |
| gpt-5.5（4月下旬〜） | 1.29億 | 40万 | 95% | $349 |
| **合計** | **約1.97億** | **約124万** | **93%** | **~$455** |

Codexのinputの特徴として、**93%がcached input**だった。モデルが新しいほどキャッシュ比率が高く、gpt-5.5では95%に達する。

モデルの進化が速く、3月のgpt-5.3-codexから始まり、4月頭にgpt-5.4、4月末にはgpt-5.5へ切り替わった。

## Claude Code vs Codex の規模感

**Claude Codeが約20倍のスケール**で消費している。

この差を生んでいる要因のひとつは**用途の違い**だ。自分の場合、Claude Codeはビジネス系タスク（ドキュメント作成・調査・設計・資料整理など）に使っていて、Codexは実装（コーディング）に特化している。ビジネス系のタスクはコンテキストが大きくなりやすく、会話が長くなる分だけトークンも積み上がる。

設計的な違いも大きい。Claude Codeは対話型で1セッションあたりのコンテキストが巨大（CLAUDE.md + コードベース + 会話履歴）。一方Codexはタスク単位で完結するため1セッションのトークン消費は比較的コンパクト。

ただしCodexのセッション数108に対してClaude Codeは同期間で384セッション。セッション単位で割ると差はもう少し縮まる。

## 所感

- **4月の爆発**はMulti-agent並列実行を本格的に試した時期と一致する。Subagentを10本同時に走らせるだけでトークンはあっという間に跳ね上がる
- **キャッシュ依存度が高い**ということは、大きなコンテキストを前提に設計されているということ。CLAUDE.mdや標準ドキュメントを充実させるほど効果が出る
- **Codexはモデル進化が速い**。3ヶ月で3世代変わった。gpt-5.5のcached input比率の高さはコスト的に有利
- **どちらもinput<<output**。AIが書き出すトークンの方が読み込みより圧倒的に多い。これはコーディング作業では必然かもしれない

計測してみて、「なんとなく使ってる」の解像度が上がった。特にキャッシュ比率とAgent並列実行の影響は、意識していないと見えてこない部分だった。
